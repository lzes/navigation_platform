# 机器人几乎不移动 / 不朝目标前进 — 原因分析与处理

## 可能原因概览

### 1. 静态障碍物过密或坐标系不一致（最常见）

- **现象**：周围没有行人时也不朝目标走。
- **原因**：`boxes_2d.json` 中的障碍物与机器人/目标使用**同一坐标系**。若障碍物很密，或机器人/目标与障碍物坐标系/尺度不一致（例如客户端用米、障碍物用厘米），RVO2 会认为机器人被障碍物包围，ORCA 可行速度域被压成空集或只含 (0,0)，输出速度长期为 0。
- **处理**：
  - 在 `tcp_server.py` 中设 `USE_STATIC_OBSTACLES = False`，暂时不加载静态障碍物，验证“无障碍时是否能正常朝目标移动”。
  - 确认 `boxes_2d.json` 与客户端上报的 `robot_pos`、`robot_target` 使用**相同单位与坐标系**；必要时做坐标变换后再传给 ORCA。

### 2. 禁止后退 + ORCA 只给出向后速度

- **现象**：有行人或障碍时更容易停住。
- **原因**：`allow_backward = False` 时会把“相对目标的后退分量”去掉，只保留侧向。若 ORCA 给出的安全速度主要是向后，投影后可能接近 (0,0)。
- **处理**：已在 ORCA 后处理中增加 **min_creep_speed**：当“未到目标”且当前输出速度小于该值时，强制为沿目标方向的最小速度（默认 0.15），避免长期为 0。

### 3. ORCA 可行域为空（行人/障碍过多）

- **现象**：人多或障碍多时几乎不动。
- **原因**：RVO2 的线性规划在约束下找最接近 pref_vel 的速度；约束过紧时可行域为空或只含原点，会得到 (0,0)。
- **处理**：同上，用 **min_creep_speed** 在“未到目标”时兜底，保证至少以最小速度朝目标前进。

### 4. time_step 未设置

- **现象**：创建仿真时报错或行为异常。
- **原因**：`policy.time_step` 为 `None` 时，建 RVO2 仿真会传 `None`，可能报错。
- **处理**：ORCA 中建仿真时使用 `time_step = self.time_step if self.time_step is not None else 0.25`，保证始终有合法步长。

### 5. 客户端数据问题

- **现象**：一直收 0.0,0.0 或几乎不动。
- **可能**：
  - `robot_target` 与 `robot_pos` 相同或非常接近（如距离 < 0.2m），服务端会直接回 0,0 视为“已到目标”。
  - 数据格式错误（缺少字段、分隔符错误），解析失败或得到错误位置。
- **处理**：确认 TCP 数据格式为  
  `robot_pos_x,robot_pos_y & robot_vel_x,robot_vel_y & robot_target_x,robot_target_y & [行人...]`  
  且目标与当前位置在“未到达”时距离 ≥ 约 0.2m。

---

## 已做代码修改摘要

1. **ORCA (`orca.py`)**
   - 建仿真时：`time_step` 为空则用默认 `0.25`。
   - 后处理：在返回前统一做“未到目标且速度过小”的检查，若 `dist > goal_radius` 且速度模长 < `min_creep_speed`，则强制为 `min_creep_speed * (to_goal/dist)`，保证未到目标时至少以最小速度朝目标走。

2. **tcp_server (`tcp_server.py`)**
   - 增加 `USE_STATIC_OBSTACLES`（默认 `True`）。排查时改为 `False` 可关闭静态障碍物，用于验证是否为障碍物导致不动。

---

## 建议排查顺序

1. 将 `USE_STATIC_OBSTACLES` 设为 `False`，看无静态障碍时是否正常朝目标移动。
2. 若仍不动，检查客户端发送的 `robot_pos`、`robot_target` 是否同坐标系、单位是否正确，且未到目标时距离是否大于约 0.2m。
3. 确认服务端已调用 `policy.set_time_step(0.25)`（或你使用的步长）。
4. 若希望“更容易动起来”，可适当增大 `orca.py` 中的 `min_creep_speed`（如 0.2～0.25）。
